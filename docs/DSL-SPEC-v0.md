# Kalita DSL — Specification v0 (MVP)

Status: draft. Scope: only what is included in the 8-week MVP. Keywords are in English (decision HLD §10). Files: `*.dsl`, encoding UTF-8. Pack = directory with `pack.dsl` (manifest) + module files.

## 0. Grammar philosophy

1. **Narrowness is the source of guarantees.** Every construct is either fully checkable by the compiler, or it does not belong in the grammar.
2. **One way to express one thing.** No synonyms or sugar — smaller drift surface when generated by agents.
3. **Errors are written for agents:** every compilation error = `{code, file:line, message, fix_hint}` — machine-readable, with a fix hint (agent self-correction loop).
4. All names — `snake_case` for fields, `PascalCase` for entities/states/roles.

## 1. Pack manifest

```
pack collections
version 0.1.0
requires kalita >= 0.1
depends core >= 0.1        # base pack: User, attachments, notifications
```

## 2. Data types (closed list v0)

| Type | Note |
|---|---|
| `string` | up to 1 KB, indexable |
| `text` | unlimited, full-text |
| `int`, `float`, `money`, `bool` | money stores the currency |
| `date`, `datetime` | TZ-aware |
| `enum[A, B, C]` | closed list of values |
| `ref[Entity]` | + `on_delete = restrict \| set_null \| cascade` |
| `array[ref[Entity]]` | references only in v0 |
| `file` | attachment (stored by the core) |

Not in v0: `json`, geo, configurable decimal precision, nested structures.

## 3. Entities

```
entity Debtor:
    company: string required
    contract: ref[Contract] on_delete=restrict
    debt: money
    overdue_days: int computed = days_since(contract.due_date)
    status: enum[OnTime, Overdue, Claim, Legal, Settled] default=OnTime
    manager: ref[core.User] default=$me

constraints:
    unique(company, contract)
```

Field modifiers: `required`, `default=<expr>`, `unique`, `computed=<expr>` (read-only, recalculated by the core). Every entity automatically receives: `id`, `created_at/by`, `updated_at/by`, full change history (event store) — no declaration needed.

## 4. Workflow

```
workflow Debtor on status:
    OnTime  -> Overdue: auto when overdue_days > 0
    Overdue -> Claim:   send_claim assignee=agent(Collector)
    Claim   -> Legal:   escalate requires approval(FinDirector)
    any     -> Settled: auto when debt = 0
```

Transition semantics: `Source -> Target: action_name [auto] [when <guard>] [assignee=<role|agent(Role)|auto>] [requires approval(<Role>)]`.

- `auto` — executed by the core when the guard evaluates to true;
- `assignee` — who gets the task to execute the transition;
- `requires approval(Role)` — HITL: the transition creates an entry in the signing queue; **the transition does not exist without the signature**. The signature is written to the log cryptographically.
- The compiler checks: reachability of all states, absence of transitions to/from non-existent states, conflicts among auto-guards.

## 5. Roles and permissions

```
roles:
    Accountant
    FinDirector
    Collector agent          # agent marker: the role is fulfilled by an agent

permissions:
    Collector:
        read  [Debtor, Contract]
        act   [send_claim]
        deny  [update Debtor.debt, delete *, read Contract where classified = true]
    Accountant:
        full  [Debtor]
        read  [Contract]
    FinDirector:
        approve [escalate]
        read    all
```

- Actions: `read | create | update | delete | act [transition_names] | approve [names] | full`.
- Row-level: `where <expr>` (e.g. `read Debtor where manager = $me`).
- Field-level: `deny update Debtor.debt`.
- **Deny is stronger than allow. For agent roles the deny block is mandatory** (the compiler requires explicit denials — an agent without explicit boundaries does not compile).
- A subject with no rule = no access (default deny).

## 6. Automation

```
automation:
    on schedule daily at 09:00 for Debtor when status = Overdue and overdue_days in [3, 7, 14]:
        agent Collector: draft_reminder(tone = soft if overdue_days < 7 else firm)
        notify email(manager)

    on update Debtor when status changed to Legal:
        webhook out "https://legal.example.com/intake"

    on stuck Debtor in Claim for 10d:
        escalate_to FinDirector
```

Triggers v0: `on schedule <schedule> [for <Entity>] [when ...]` (with `for` — the rule runs over entity records; without it — a global action), `on create|update|delete <Entity> [when ...]`, `on stuck <Entity> in <State> for <duration>`.
Actions v0 (closed list): `agent <Role>: <task>(args)` (create a task for an agent), `notify email(...)`, `webhook out <url>` (outbound only, declared = visible in the spec), `create/update <Entity> set {...}`, `escalate_to <Role>`.

## 7. Expressions (intentionally limited)

- Literals, field paths (`contract.due_date` — one ref hop), `$me`, `now()`, `today()`, `days_since(x)`;
- Comparisons, `in [...]`, `and/or/not`, inline `<a> if <cond> else <b>`;
- Closed list of functions. **No:** loops, aggregates (v0), user-defined functions, outbound calls from expressions.

## 8. UI

```
ui Debtor:
    list: [company, debt, overdue_days, status] sort=-overdue_days
        filters: [status, manager]
        view "My debtors": where manager = $me
    form:
        section "General": [company, contract, debt]
        section "Status":  [status, overdue_days]
    board: by status
```

Built-in without declaration: signing queue (approval inbox), object journal, global search. Not in v0: dashboards/charts, custom widgets, themes.

## 9. Migrations (v0 rules — additive only)

Allowed by diff: new entity; new field (nullable or with default); new enum value (appended at end); new state/transition; new role/permission; new automation; any UI changes.
Forbidden in v0 (manual procedure with export required): changing a field type, deleting/renaming a field or entity, deleting an enum value or state.
Each accepted diff = a new SystemDefinition version; application is atomic; rollback = reverse proposal through the same signing queue.

## 10. Reserved words (not implemented, but forbidden as identifiers)

`component` (escape hatch/WASM), `integration`, `simulate`, `dashboard`, `i18n`, `metric`, `system` (system packs), `federation`/`remote` (cross-node references) — so that v0 packs do not conflict with v1.

## 10a. System packs (kalita-2024 heritage, v1 plan)

The platform core is described as much as possible in kalita itself: `core` is a pack with
`core.User`, reference data (master data), and attachments. The difference from ordinary
packs is the `system` marker: its contents can only be changed by a platform update;
the ordinary change pipeline rejects such changes (protection, not privacy). Principle:
everything that CAN be a pack MUST be a pack; only what cannot be a pack remains in the
core (log, permissions, compiler, runtime).

## 10b. Node federation (plan, layer 3)

One node = one perimeter, but nodes can interoperate: a pack can declare an external
reference to an entity in another instance (two domain areas — two nodes — one
protocol). Reserved: `remote[node.Entity]` as a reference type; exchange —
typed contracts via an MCP bridge with the identity of both nodes and a log on
both sides. Not implemented until layer 3.

## 12. What is deliberately absent from v0 — and why

| Absent | Why | When |
|---|---|---|
| Aggregates/reports | heavy semantics, does not block the first packs | v1 |
| Escape hatch (component) | WASM contract requires a separate design | v1 |
| Inbound integrations | perimeter security; outbound webhooks are sufficient | v1 |
| Cross-entity computations deeper than 1 ref | explodes compiler complexity | v1 |
| Destructive migrations | risk #1 of the project (HLD §9) | v2 |
