# ADR-002: Многоуровневое хранение — журнал один, проекций много

**Status:** Accepted (2026-06-13). Дополняет ADR-001 (его условие пересмотра
срабатывает здесь управляемо, а не аварийно).

**Context:** платформа должна выдерживать большие массивы событий — уровень
«взрослого» enterprise event sourcing, не уровня ClickHouse, но с честными
проекциями. Вопрос фаундера: можно ли разные базы для разных модулей.

**Decision — четыре правила:**

1. **Журнал один и только в PostgreSQL.** Истина не размножается. Масштаб:
   партиционирование по месяцам (запланировано), индекс seq, append-only —
   PG спокойно держит сотни миллионов событий. Снапшоты головы проекций +
   реплей хвоста — для быстрого старта при больших журналах (v1).
2. **Проекции ядра — двухъярусные.** Горячие — в RAM (как сейчас); сущность,
   перерастающая порог (~1M записей или бюджет старта из ADR-001), переезжает
   в SQL-проекцию PG (jsonb-таблица, обновляется в той же транзакции, что и
   append — исходный путь HLD §3.4). Журнал не меняется, поведение не
   меняется, меняется только носитель проекции.
3. **Разные базы — да, но на уровне модулей-воркеров, не ядра.**
   `Store.Since(seq)` — это публичный механизм подписки: любой воркер строит
   СВОЮ проекцию в СВОЕЙ базе (Qdrant у knowvault — уже так; аналитика →
   DuckDB/ClickHouse; полнотекст → whatever). Правила собственности: модульная
   база (а) наполняется только реплеем журнала, (б) выбрасываема — теряется
   без потери истины, (в) принадлежит воркеру, ядро о ней не знает.
4. **Журнал = внутренняя шина.** Никаких отдельных брокеров (Kafka и т.п.) в
   v0/v1: водяные знаки по seq дают exactly-once потребителям; федерация
   (HLD §8a) ездит на том же механизме.

**Alternatives considered:** (1) сразу SQL-проекции для всего — отвергнуто:
сложность миграций проекций раньше времени, RAM-ярус покрывает 95% паков;
(2) встроить ClickHouse для аналитики в ядро — отвергнуто: чужая ниша,
правило 3 решает это воркером; (3) Kafka как шина — отвергнуто: журнал уже
является надёжной упорядоченной шиной, второй брокер = вторая истина.

**Expected outcome (falsifiable):** узел с 10M событий и 1M записей одной
сущности: старт < 30 сек (со снапшотом), p95 чтения < 50ms, запись — сотни
соб/сек. Не проходит — пересматриваем ярус 2 (шардирование проекций).
