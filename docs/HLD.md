# Kalita — High-Level Design

Версия 0.1 (2026-06-12). Статус: проект для обсуждения.

Kalita — исполняемый рантайм бизнес-систем для агентной эпохи: агенты и люди описывают систему на ограниченном DSL, рантайм исполняет описание (никакой генерации кода наружу), каждое изменение — подписанный diff, каждое действие — событие в журнале.

---

## 1. Принципы (конституция архитектуры)

1. **Спека исполняется, а не генерирует код.** Сгенерированный код как внешний артефакт запрещён (урок MDA/CASE: дрейф спеки и кода убивает источник истины).
2. **Лего, не сварка.** Всё, что не выразимо грамматикой DSL, не существует. Расширение возможностей — через компоненты с типизированным контрактом (escape hatch), не через расширение грамматики «до можно всё».
3. **Event sourcing в ядре.** Истина — append-only журнал событий; текущее состояние — проекция. Без этого невозможны: симуляция перед подписью, time travel, репутация агентов, страхуемость (слои 2–3).
4. **Подпись — первоклассная криптографическая сущность.** Кто, что, когда, на основании чего. Журнал должен иметь доказательную силу.
5. **Агент = сотрудник.** У агента есть identity, должность (роль), полномочия, рейтинг и журнал. Рантайму безразлично, чем агент думает (Claude/LangGraph/прочее) — рантайм ограничивает, что агент может сделать.
6. **Мышление крупно, коммиты мелко.** Любое изменение системы — маленький diff DSL через конвейер: валидация → симуляция → подпись → применение с миграцией.
7. **On-prem first.** Один бинарь/контейнер, данные не покидают контур. Облако — вариант деплоя, не архитектуры.

## 3. Компоненты

### 3.1 DSL Layer
- **Грамматика** (EBNF, версионируемая): `entity`, `workflow`, `права`, `автоматизация`, `ui`, `пак` (модуль). Минимальная по принципу №2.
- **Паки** — единица дистрибуции: набор DSL-файлов + версия + зависимости + манифест. Формат пригоден для маркетплейса с первого дня (semver, подпись автора).
- Грамматика публикуется в машиночитаемом виде (для constrained decoding: агент генерирует DSL под грамматикой — невалидное невыразимо на уровне токенов).

### 3.2 Compiler / Validator
- Parse → типизированная модель (AST → Semantic Model).
- Проверки: ссылочная целостность (все `ref`, роли, переходы существуют), права непротиворечивы, workflow достижим/без тупиков.
- Выход: `SystemDefinition vN` + **MigrationPlan** (diff между vN-1 и vN на уровне схемы и данных).

### 3.3 Runtime Kernel
- **Entity Engine:** CRUD над проекциями, валидация, computed-поля, ссылочная целостность. Хранение: PostgreSQL (jsonb + индексируемые колонки), проекции перестраиваемы из событий.
- **Workflow Engine:** состояния, переходы, guards, `assignee` (человек/агент/авто), `требует=подтверждение(роль)` — HITL-переход создаёт задачу-подпись.
- **Permission Engine:** RBAC + ABAC (роль, владение, поле-уровень, гриф). Единая точка: каждое действие любого субъекта (человек, агент, автоматизация) проходит через неё. Запреты выразимы явно (`запрещено [...]`).
- **Automation Engine:** триггеры (`when`-условия), расписания, SLA/эскалации, вызов агентов как исполнителей шагов.

### 3.4 Event Store + Signatures
- Append-only лог: `{event_id, ts, actor (human|agent|system), action, payload, basis (правило/подсказка/ADR), signature?}`.
- Подпись: Ed25519, ключи субъектов; подписываются изменения DSL и HITL-решения. Цепочка хэшей по журналу (tamper-evident).
- Из событий строятся: текущее состояние, история любого объекта, симуляции, рейтинги агентов (слой 2).

### 3.5 Change Pipeline (главный конвейер платформы)
```
Proposal(diff DSL, author, basis)
  → Validate (компилятор: зелёное/красное)
  → Simulate (опционально: прогон правила по историческим событиям → отчёт последствий)
  → Sign (роль с правом подписи; UI «очередь подписей»)
  → Apply (атомарно: новая версия SystemDefinition + миграция данных + событие)
  → Rollback (любая версия восстановима: определения версионны, данные событийны)
```
Прямое изменение системы в обход конвейера не существует как операция API.

### 3.6 Agent Gateway (MCP)
- MCP-сервер — первоклассный интерфейс (равный Web UI): tools `describe_system`, `query/mutate entities`, `propose_change(diff)`, `take_task / complete_task`, `read_journal`.
- Каждый агент: identity + роль + scopes + rate limits + рейтинг. Анонимных агентов нет.
- Mutate-операции агента = те же права Permission Engine, что у человека в этой роли.

### 3.7 UI (генерируемый)
- Из `ui`-секций DSL: списки (фильтры, сортировки, views), формы (секции, виджеты по типам), доски по статусам, **очередь подписей** (главный экран человека), журнал объекта.
- Реализация: SPA (React) поверх Meta API; никакой ручной вёрстки на приложение. Кастомные виджеты — через escape hatch.

### 3.8 Escape Hatch (компоненты)
- Внешние функции/компоненты с типизированным контрактом (вход/выход по схеме, декларация side-effects), вызываются из DSL (`agent:`-шаги, computed, интеграции, кастомные виджеты).
- Компонент не может: обойти Permission Engine, писать в store напрямую, менять определения. Исполнение — sandboxed (отдельный процесс/WASM — решить в детальном дизайне).

## 4. Ключевые потоки

**A. Создание системы:** агент интервьюирует/читает Excel → `propose_change(полный DSL)` → валидация → человек смотрит DSL+превью UI → подпись → рантайм разворачивает (таблицы, API, UI) — минуты.

**B. Изменение:** «не дёргать VIP» → агент шлёт diff в 3 строки → симуляция: «за 90 дней затронуло бы 12 заявок» → подпись → применено с миграцией.

**C. Агент-работник:** триггер `when дней_просрочки=14` → задача агенту → агент действует в рамках scopes → событие в журнал с basis → критичный переход создаёт подпись человеку.

**D. Откат:** любое применённое изменение — `revert` как новый proposal (тоже через подпись).

## 5. Технологический стек (предложение)

| Слой | Выбор | Почему |
|---|---|---|
| Ядро | **Go** | лучший в классе single static binary; язык open-source инфраструктуры (Kubernetes, Terraform, Grafana, Temporal) → максимальный пул контрибьюторов; малый footprint для on-prem; простой язык = меньшая поверхность дрифта для агентной генерации кода. Опыт фаундера в C# сознательно НЕ критерий: код пишет конвейер агентов, язык выбирается под экосистему |
| Хранилище | PostgreSQL 16 | событийный лог + jsonb-проекции + полнотекст; единственная зависимость |
| Парсер DSL | собственный, ручной рекурсивный спуск | грамматика — ядро IP; нужен идеальный контроль ошибок и сообщения для агентов |
| Sandbox (escape hatch) | WASM (wazero, pure-Go) | детерминизм, capability-based, кроссплатформенно (включая Windows), language-agnostic компоненты, готово к маркетплейсу недоверенных паков. Процессная изоляция отвергнута: непереносима |
| UI | React + Meta API | генерация из метаданных |
| MCP | в ядре (HTTP + stdio) | первоклассный интерфейс |
| Деплой | Docker Compose (kalita + postgres); один клиент = один узел | on-prem за 10 минут; изоляция «ваш узел — ваш контур» вместо shared-схемы |

## 6. Безопасность
- Субъекты: люди (OIDC/локальные), агенты (ключи), автоматизации. Все действия — через Permission Engine, все — в журнал.
- Гриф на сущности/поля (`запрещено [видеть Договоры с грифом]`).
- Секреты интеграций — в локальном vault ядра, не в DSL.
- Сеть: один входной порт; исходящие — только декларированные в DSL интеграции (для контура важно: «что наружу» видно из спеки).

## 7. Деплой и масштаб
- MVP: single node, вертикальный рост; целевые системы — десятки-сотни пользователей на узел (профиль 1С, а не Netflix).
- Позже: read-реплики проекций, перестройка проекций из событий = бесплатная миграция/восстановление.

---

## 8. Этапность

### MVP (8 недель) — слой 1
Входит: грамматика v0 (entity, workflow, права, автоматизация-минимум, ui-минимум), компилятор + миграции добавления (без разрушающих), Entity/Workflow/Permission engines, Event Store **с подписями (day-1!)**, Change Pipeline без симуляции, MCP-сервер, генерация UI (список/форма/доска/очередь подписей), Docker.
Не входит: симуляция, аналитика/дашборды, i18n, маркетплейс, кросс-орг, escape hatch (заглушка-контракт), разрушающие миграции (только через ручную процедуру).

### Слой 2 (6–18 мес): симуляция перед подписью, рейтинги агентов, библиотека паков с телеметрией выживаемости, детектор дрифта на корпусе подписей.

### Слой 3 (18+ мес): кросс-организационные контракты, переносимая репутация агентов, рынок подписантов, страхуемость.

**Два необратимых решения дня №1, защищающих слои 2–3:** event sourcing как единственная истина; криптоподпись в схеме журнала. Стоят страницу кода сейчас — невозможны задним числом.

## 8a. Масштабирование и федерация (принцип, 2026-06-13)

Один узел = одна доменная область одной компании (bounded context): запись —
сотни событий/сек (сериализована advisory-lock), чтение из RAM-проекций,
старт — реплей (бюджет ADR-001). Для ICP — с многократным запасом. Шардинг
одного домена не делается никогда; HA — PG-реплика + standby-реплей (v1).

Рост = дробление по доменам: «микросервисы из калит». Журнал — готовый outbox
(seq + идемпотентность = exactly-once); мост между узлами — агент с identity и
deny-границами с обеих сторон; обмен схемами — паки через change pipeline
(дрейф контрактов невыразим). **Запрет: никаких синхронных RPC между узлами —
только асинхронные события через мост-агента.** Зарезервировано:
`remote[node.Entity]` (DSL-SPEC §10b), слой 3.

## 9. Главные риски (честно)
1. **Движок миграций** — самая сложная инженерия проекта (не парсер, не UI). Ограничить MVP аддитивными миграциями.
2. **Потолок выразительности DSL** — лечится escape hatch, не ростом грамматики; дисциплина решает судьбу проекта.
3. **Качество генерируемого UI** — «достаточно хорошо» для учётных систем достижимо; пиксель-перфект не цель.
4. **Окно:** Tessl/Kiro/Supabase в одном шаге от ниши. Скорость > полнота.
5. **Соло-мейнтейнер** — bus factor 1; смягчение: open core + ранние контрибьюторы.

## 10. Решённые вопросы детального дизайна
- **Синтаксис DSL: только английские ключевые слова.** Двуязычность отвергнута: два диалекта = дрифт и раскол паков; международный рынок и обучающие данные моделей — английские. Локализация лейблов/сообщений — слой i18n, не грамматика.
- **Sandbox: WASM** (см. стек). Заглушка в MVP, но контракт компонентов проектируется под WASM сразу.
- **Мультитенантность: один клиент = один узел** (MVP и облако: узел-на-тенанта, не shared-схема — проще и честнее по безопасности).

Открытым остаётся: формат симуляции (полный реплей vs выборочный по затронутым правилам) — решить по факту производительности event store.
