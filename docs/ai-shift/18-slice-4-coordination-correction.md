# Slice 4 correction: general coordination layer

## 1. Why `DailyPlan` is too narrow as a core concept

`DailyPlan` is a useful departmental artifact, but it is too narrow to become the mandatory runtime abstraction between `WorkItem` backlog and execution.

### Why it is too narrow
- It hardcodes one coordination cadence: day-based planning.
- It assumes execution must be pre-arranged in a dated batch before work can run.
- It treats manager-style planning as universal, even though some departments operate continuously from queues.
- It leaves no neutral place for future machine coordination, such as a digital manager that reprioritizes work dynamically.
- It pushes SLA and urgency semantics into a planning artifact instead of keeping them available for multiple coordination modes.

### Architectural consequence
If `DailyPlan` becomes the core transport-free object, Kalita will accidentally encode one operating mode into the runtime center. That would make continuous execution, human gating, digital-manager routing, and SLA-first prioritization look like exceptions instead of first-class modes.

### Correction
Slice 4 should introduce a general coordination layer, not a planning-only layer. `DailyPlan` should be treated later as one optional coordination product generated by that layer.

## 2. Proposed replacement abstraction

### Recommended name
`CoordinationDecision`

Alternative acceptable names:
- `WorkCoordinationDecision`
- `CoordinationDirective`
- `ExecutionCoordinationDecision`

`CoordinationDecision` is preferred because it is broad enough for batch planning, queue release, manager gating, and future digital-manager choices without implying a transport, UI, or schedule format.

### Purpose
`CoordinationDecision` is the minimal durable record that says **how a `WorkItem` may progress from backlog into execution readiness**.

It does not describe tool steps and it does not require a dated plan. It is a transport-free operating decision that sits between:
- upstream: `Case` / `WorkItem` backlog state
- downstream: execution start or execution eligibility

### Minimal fields
Keep the first version intentionally small.

- `ID string`
- `WorkItemID string`
- `CaseID string`
- `Mode string`
  - examples: `continuous`, `batch_plan`, `manager_gate`, `digital_manager`, `sla_push`
- `Decision string`
  - examples: `ready`, `hold`, `assign`, `defer`, `escalate`
- `AssigneeRef string`
  - optional; human, team, queue slice, or digital employee reference
- `Priority int`
  - normalized execution ordering signal independent of any one mode
- `ReasonCodes []string`
  - compact explanation such as `sla_due_soon`, `manager_override`, `awaiting_review`
- `NotBefore *time`
  - optional delay / release gate for future schedulers
- `ExpiresAt *time`
  - optional window boundary for stale coordination decisions
- `DecidedBy ActorRef`
  - human manager, system policy, or future digital manager identity
- `CreatedAt time`

### Lifecycle role
`CoordinationDecision` should play one narrow role:
1. observe a `WorkItem` in backlog or queued state
2. record the current coordination outcome for that work item
3. make execution either eligible, deferred, assigned, or blocked
4. allow later coordination modes to supersede or refresh the decision without rewriting the execution model

In other words, Slice 4 should define the **execution-readiness boundary**, not a day planner.

## 3. Interfaces required for Slice 4

Introduce only the smallest transport-free interfaces needed to make coordination a separable layer.

### `CoordinationService`
Purpose: evaluate backlog work and produce or update `CoordinationDecision` records.

Minimal responsibilities:
- accept a `WorkItem` plus coordination context
- produce a new decision or refresh an existing one
- keep decision production independent of HTTP, cron, or queue transport

### `CoordinationRepository`
Purpose: persist and query decisions.

Minimal responsibilities:
- store a decision
- fetch latest active decision for a `WorkItem`
- list decisions by queue / assignee / mode for read models and later adapters

### `CoordinationPolicy` or `CoordinationStrategy`
Purpose: encapsulate the rule that determines the next coordination outcome.

Minimal responsibilities:
- read `WorkItem` state, case context, and basic capacity / urgency signals
- return a transport-free recommendation such as `ready`, `hold`, `assign`, or `defer`

### `ExecutionEligibilityGate`
Purpose: provide the narrow integration seam into Slice 5 execution later.

Minimal responsibilities:
- answer whether a `WorkItem` may start execution based on its latest coordination decision
- avoid coupling execution start directly to queue presence or plan membership

## 4. How it integrates with slices 1-3

This correction does **not** redesign slices 1-3. It only adds a more general layer after them.

### Slice 1: events
No change to Slice 1. Events still capture business facts and runtime facts.

### Slice 2: commands
No change to Slice 2. Commands still represent explicit intent and command admission remains the action boundary.

### Slice 3: cases, queues, work items
No change to Slice 3's purpose. Slice 3 should still create:
- `Case`
- `WorkQueue`
- `WorkItem`

The correction is only what happens **after** `WorkItem` exists.

Updated handoff:
- Slice 3 ends with backlog or queue intake.
- Slice 4 adds coordination decisions over that backlog.
- Slice 5 later starts execution from eligible coordinated work.

So the flow becomes:
`Event -> Case -> WorkItem -> CoordinationDecision -> Execution`

That preserves slices 1-3 while removing the requirement that every item must enter a day plan.

## 5. How future modes can be built on top

The point of the new abstraction is that many modes can generate the same core decision type.

### `DailyPlan` as an optional mode
A later planning module can build a dated plan artifact for one team or employee. That artifact can then emit `CoordinationDecision` records with:
- `Mode = batch_plan`
- `Decision = assign` or `ready`
- optional assignee and planning rationale

This keeps `DailyPlan` useful without making it universal.

### Continuous execution mode
A continuous queue-driven department can skip plan generation entirely. A lightweight coordination strategy can issue:
- `Mode = continuous`
- `Decision = ready`

Execution can then pull from coordinated-ready items according to priority and guardrails.

### Digital manager mode
A future digital manager can inspect backlog, queue pressure, and case signals, then write:
- `Mode = digital_manager`
- `Decision = assign`, `defer`, `escalate`, or `hold`

Because the core object is neutral, machine-made coordination does not require a new runtime abstraction.

### SLA mode
An SLA-oriented coordinator can monitor due times and urgency windows, then emit decisions such as:
- `Mode = sla_push`
- `Decision = ready` or `escalate`
- `ReasonCodes` including `sla_due_soon` or `breach_risk`

This lets SLA pressure influence execution eligibility without pretending that every decision came from a daily plan.

## 6. Minimal migration / correction plan from the current Slice 4 direction

Keep the correction incremental.

### Step 1
Stop describing Slice 4 as `DailyPlan` generation in core architecture documents.

### Step 2
Rename the Slice 4 target from planning/assignment to **coordination decisions and execution eligibility**.

### Step 3
Keep any existing queue and assignment ideas, but restate them as optional outputs of coordination rather than mandatory runtime prerequisites.

### Step 4
Move `DailyPlan` out of the minimum core model for Slice 4 and place it in a later optional mode or extension layer.

### Step 5
Ensure Slice 5 execution starts from `WorkItem` records that are marked eligible by coordination, not from membership in a dated plan.

## 7. Recommendation

### Core recommendation
Replace `DailyPlan` completely in Slice 4.

Slice 4 should introduce:
- `CoordinationDecision`
- a minimal coordination service / repository
- an execution eligibility gate

### What to do with `DailyPlan`
Keep `DailyPlan` only as a later optional artifact built on top of the coordination layer.

That gives Kalita the right long-term shape:
- backlog and queue semantics remain in slices 1-3 and the existing operational model
- Slice 4 stays transport-free and mode-neutral
- future continuous, manager-driven, digital-manager, and SLA-driven coordination all fit without re-architecting the runtime
- day planning remains supported, but only as one coordination mode rather than the universal core abstraction
