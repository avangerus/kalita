## ПЛАН ДОРАБОТОК ЯДРА (по критичности, с учётом факта)

### Блокеры — без них приложения не «настоящие»
1. **Б1. Comment как примитив** (core-пак + UI-секция timeline). Полиморфная
   привязка к любой записи, is_internal видимость по ABAC, нативный тред в
   detail-view. Нужен ВСЕМ трём. Самый важный.
2. **Б4. Арифметика в computed**: `amount * probability / 100`, `limit - used`.
   Расширить язык выражений: +,-,*,/ между полями одной записи. Нужен CRM
   (forecast) и HR (баланс). Язык условий уже есть — добавить арифметику в
   computed-вычислитель.
3. **Б5. null в языке выражений**: `where certificate != null`, `field = null`.
   Условные переходы по заполненности. Мелко, но блокер HR.
4. **Б6. report/dashboard-блок**: сводки group-by с агрегатами по ВСЕМ записям
   (не только `where reffield=$self`). On-prem, из ядра. Нужен CRM (воронка),
   HR (метрики), и это Dynamics-обвязка.
5. **Б3. array[file]**: несколько вложений. Desk (скриншоты+логи), HR (резюме).
6. **Б7. hours_since/minutes_since**: SLA < суток. Desk.

### Важно — приложение неполноценно без этого
7. **calendar-view**: график отпусков (HR), активности (CRM). Новый view-тип.
8. **бизнес-календарь** (data-пак рабочих дней + business_days в duration/stuck):
   юридически корректные расчёты HR, SLA Desk.
9. **datetime-триггер** `on Activity.remind_at approaching`: персональные
   напоминания (CRM).
10. **core.User как настоящая системная сущность** + i18n лейблов (RU).

### Желательно
полиморфные ссылки, свимлейны, saved-фильтры рантайма, in-portal уведомления.

## ⭐ Opus magnum: Jira, где задачи делают агенты (мечта фаундера)
«Джира, где задачи делают агенты, и ты можешь ставить им задачи». Это не новый
пробел — это КОМБИНАЦИЯ всего, что мы строили, в одном приложении. Уже есть
почти всё: трекер-пак (задачи/связи/воронка) ✅, агенты как assignee=agent ✅,
задачи-в-пул для агентов ✅, MCP (агент берёт задачу, работает, отчитывается) ✅,
HITL-подпись на мёрж ✅, dev_department-пак (отдел разработки на kalita) ✅,
dogfood (kalita разрабатывает себя через себя) ✅.
Чего не хватает до «поставил задачу → агент сделал → ты принял» как ПРОДУКТА:
(а) Comment (Б1) — диалог с агентом в задаче; (б) report/dashboard (Б6) — видеть
прогресс; (в) надёжный agent-runner воркер (Claude Code как сервис, берущий
задачи из пула и реально кодящий/делающий) — это воркер, не ядро; (г) UI
«поставить задачу агенту» — кнопка, создающая Task с assignee=agent.
Вывод: opus magnum = трекер-пак + Comment + agent-runner воркер. Ядро почти
готово; это СБОРКА, не новый фундамент. Делать ПОСЛЕ блокеров Б1/Б4/Б6 — тогда
он соберётся как пак+воркер, доказав всю модель разом. Это и есть финальный
dogfood: kalita, управляющая своей разработкой агентами, как продукт.
