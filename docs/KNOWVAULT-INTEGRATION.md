# KnowVault × Kalita — интеграционный контракт

Статус: проект (2026-06-13). KnowVault — периметровый RAG-поиск по корпоративным
данным (файлы, почта, репозитории, базы, чаты). Существующая реализация:
D:\work\knowvault_app (Python: auth, workspaces, ingest, search_rag, graph,
audit; Postgres + Qdrant; React UI; docker-compose).

## 1. Решение о форме

KnowVault **не** вливается в ядро kalita и **не** переписывается на Go.

- Ядро kalita — узкая исполняемая среда с гарантиями; индексация и эмбеддинги
  раздули бы его на порядок (анти-принцип «лего»).
- KnowVault уже существует как рабочий стек со своим жизненным циклом
  (коннекторы будут меняться постоянно — ядро меняться не должно).
- Форма: **сосед-сервис в одном контуре**. Деплой: общий docker-compose
  (kalita + postgres + knowvault-стек) — «данные не покидают периметр»
  выполняется топологией, а не обещанием.

Стратегическая роль (insights 2026-06): KnowVault — не standalone-продукт
(Onyx бесплатен, ад коннекторов), а **слой данных под вертикальных агентов**:
каждому паку (дебиторка, договоры, поддержка) нужен поиск по периметру.

## 2. Разделение властей

| Вопрос | Власть |
|---|---|
| Кто этот актёр (человек/агент), какая роль | **kalita** (identity, Ed25519/WebAuthn) |
| Что актёру можно видеть | **kalita** (permission engine) + workspaces KnowVault |
| Где лежит и как ищется контент | **KnowVault** (ingest, Qdrant, RAG) |
| Журнал «кто что искал и что нашёл» | **kalita** (event store, подписи) |

Принцип: KnowVault не заводит собственных пользователей для агентов —
актёры kalita пробрасываются в KnowVault服 с ролью; маппинг роль → workspaces
конфигурируется на стороне интеграции.

## 3. Контракт поиска (MCP-tool kalita, проксируется в KnowVault)

### `search_perimeter(query, sources?, limit?)`
- Доступен агентам с ролью, которой выдано право (новый verb в DSL не нужен:
  моделируется обычной сущностью `SearchScope` или конфигом узла — решить при
  реализации; v0: конфиг узла role → workspaces).
- Каждый вызов = событие журнала kalita: `{actor, query, workspaces,
  result_count}` (kind: task.progress над служебной задачей либо новый kind
  `search.performed` — добавить аддитивно).
- Результаты фильтруются workspaces актёра ДО возврата агенту; сниппет +
  ссылка на источник; контент чувствительных источников агенту не возвращается
  целиком — только через право на чтение соответствующего источника.

### Использование из automation DSL (v1)
`agent Role: search(...)` — обычная агентная задача; агент внутри задачи
вызывает search_perimeter. Грамматика не расширяется.

## 4. Этапность

1. **Сейчас (недели 6–8 MVP kalita):** ничего не вливаем; этот документ —
   контракт. KnowVault живёт в своих папках.
2. **После MVP (этап «полигон у знакомого»):** общий docker-compose;
   search_perimeter в MCP kalita → search_rag KnowVault; проброс идентичности;
   журналирование поисков.
3. **v1:** зеркалирование прав источников (ACL почты/репозиториев ↔
   workspaces) — самая тяжёлая часть, делается по одному коннектору на
   вертикаль, по требованию платящего пака (правило субстрата).
